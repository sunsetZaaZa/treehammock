# SQL Contract Matrix for 1.0.0

This matrix is the release gate for the PostgreSQL contract in `Rigging/Database/Baseline/000_treehammock_canonical_database.sql`. It documents every stored function currently present in the baseline and the application surface that owns it. The broader 1.0.0 feature boundary is documented in `docs/release/RELEASE_FEATURE_MATRIX_1_0_0.md`, and the API surface is documented in `docs/release/API_ENDPOINT_MATRIX_1_0_0.md`.

No `CREATE PROCEDURE` routines are currently present. The 1.0.0 database surface is function-based; new stored procedures can be added later only if this matrix and the repository contract tests are updated in the same PR.

## Matrix rules

- Every `create or replace function` in the baseline must have a `### function_name` entry in this document.
- Every entry must state the owning feature, caller, inputs, returned outcomes, transaction boundary, idempotency/retry behavior, expiration/audit behavior, and required tests.
- SQL is the authority for cross-row state changes: token promotion, lockout clearing, session invalidation, audit writing, and replay prevention belong in SQL contract tests, not controller choreography.
- Backend-generated secrets and codes are passed into SQL already hashed when they are credentials/tokens. Activation codes are generated by the backend, stored by SQL, then sent to the user.

## Password reset

Password reset SQL exposes a request lookup contract plus account-bound, single-use reset artifacts and durable pending reset 2FA session state. The backend supports exactly two request delivery channels: `email` and `sms`. The reset row stores the canonical delivery channel in `method` and `delivery_channel`, with `requires_key_code = true` and `requires_totp = false`; authenticator proof lives in `pending_password_reset_sessions` after reset-token verification rather than in the request row. The backend generates a key code for the selected delivery channel, hashes that code with the configured password-reset code pepper and reset id context, computes `expires_at` from `PasswordResetSettings`, and passes the configured attempt cap into SQL. The client never sends `account_id`; finalize loads it from `password_reset_request_id`. PR Reset-8 persists reset-session selection, ordered proofs, challenge hashes, and completion state in `pending_password_reset_sessions` instead of relying on ephemeral reset 2FA cache state. PR Reset-9 revokes those pending reset sessions when authenticated 2FA method removal changes the account's eligible proof set. PR Hardening-2 removes request-strategy method names from the greenfield 1.0.0 public and SQL contracts. Request abuse controls are also backend-configured: service code builds account, destination, and IP rate-limit keys, while reset-token verification, reset 2FA proof verification, and final password promotion use separate reset-scoped abuse counters. SQL records cooldown/window counters without storing raw key codes, TOTP values, passwords, email addresses, or phone numbers. Contract coverage includes raw function-level tests plus `PasswordResetRepositorySqlIntegrationTests`, which run the real `PasswordResetRepo`, the password-reset TOTP compatibility wrappers, the shared authenticator enrollment functions, and `SessionRepo` against the canonical PostgreSQL baseline to catch mapper, parameter, replay-marker, and session invalidation drift.

#### Password reset TOTP finalize gate

PWDRESET-6 keeps TOTP verification in the backend service layer. SQL owns reset artifact state, attempt accounting, expiration, cancellation, and promotion boundaries. The backend loads a reset row through `get_password_reset_request_for_finalize`, verifies the delivery key-code hash when required, and invokes `IPasswordResetTotpVerifier` when `requires_totp = true`. PWDRESET-FIX-2D adds the protected-secret storage foundation on `two_factor_authentications`: `totp_secret_ciphertext`, `totp_secret_nonce`, `totp_secret_tag`, `totp_secret_version`, and `totp_last_used_step`. Wrong key-code or wrong TOTP calls `register_password_reset_failed_attempt`; missing TOTP is treated as request validation failure and does not mutate attempt counters. The concrete verifier fails closed on missing enrollment, unsupported provider mode, invalid protected secret material, wrong codes, and replay, so `_totp` reset methods cannot silently bypass the second proof. TOTP-RESET-2 routes verifier lookup and replay marking through the shared authenticator enrollment repository.
